A. 訂單管理
管理員 / 行政
─
▾
常規情境
A01 — 手動建立新訂單
進入訂單管理，點「新增訂單」，選擇客戶、商品、送貨日期，送出後訂單出現在列表。
/dayone/orders
A02 — 訂單列表篩選（日期、客戶、司機、狀態）
各篩選條件獨立運作，組合篩選結果正確。
/dayone/orders
A03 — 單筆換司機（pending / assigned 狀態）
訂單在 pending 或 assigned 時可點換司機，確認後列表立即更新。
/dayone/orders
A04 — 刪除訂單（pending 狀態）
確認刪除後訂單從列表消失，相關 pending_returns 未誤刪。
/dayone/orders
A05 — LIFF 訂單列表（liff-orders 頁）
LINE 下單的訂單出現在 /dayone/liff-orders，資訊正確。
/dayone/liff-orders
邊界 / 例外情境
A06 — 建訂單超過信用額度邊界
超額時出現 toast 警告，但訂單仍可送出（軟性警示，不阻擋）。
/dayone/orders
A07 — 已派車訂單不可換司機邊界
已派車狀態下換司機按鈕應灰掉或隱藏，操作無效。
/dayone/orders
A08 — 庫存不足時建訂單邊界
庫存為 0 時仍可建訂單（訂單先存），派車時才實際扣庫存，確認系統行為符合預期。
/dayone/orders
🚛
B. 派車工作台
管理員 / 行政
─
▾
常規情境
B01 — 生成派車單關鍵
選擇日期與司機，點「生成派車單」，今日訂單被合併成派車單，狀態變 draft。
/dayone/dispatch
B02 — 列印派車單有內容關鍵
點列印後開啟列印預覽，確認客戶名、商品品項、數量、地址均完整顯示，無空白。
/dayone/dispatch
B03 — 列印後庫存扣減關鍵系統自動
列印後到 /dayone/inventory 確認庫存已扣減，數字與派車單品項一致，不出現負數。
/dayone/dispatch → /dayone/inventory
B04 — 手動加站（manualAddStop）
已有派車單後補加一筆停靠點，新站出現在派車單內容中。
/dayone/dispatch
邊界 / 例外情境
B05 — 重複生成派車單防護邊界
同一天同一司機重複點「生成」，系統不應產生重複派車單（已有 draft/printed/in_progress/pending_handover/completed 狀態時排除）。
/dayone/dispatch
B06 — 無訂單日生成派車單邊界
選一個沒有訂單的日期，生成按鈕後系統不崩潰，給出合適提示。
/dayone/dispatch

C. 司機端
司機
─
▾
常規情境
C01 — 今日任務列表關鍵
司機登入後 /driver/today 顯示今日派車單所有停靠點，客戶、品項、地址正確。
/driver/today
C02 — 標記送達關鍵
進入訂單詳情，點送達，狀態更新為 delivered，AR 應收自動建立。

C03 — 現金收款關鍵

/driver/order/:id → /dayone/ar
C04 — 客戶簽名
簽名板可正常手寫，上傳後簽名圖顯示在訂單詳情（Cloudflare R2）。
/driver/order/:id
C05 — 剩貨回報（worklog）關鍵
司機在 /driver/worklog 填入剩貨品項與數量，送出後 dy_pending_returns 出現待驗紀錄。
/driver/worklog
C06 — 日結（每張派車單各自日結）
worklog 日結按鈕可依派車單分別結帳，現金、剩貨回庫數字正確。
/driver/worklog
C07 — 撿貨唯讀路線
/driver/pickup 顯示今日撿貨清單，只能查看不能修改。
/driver/pickup
邊界 / 例外情境
C08 — 無任務日進入 /driver/today邊界
今日無派車單時顯示空狀態提示，不崩潰。
/driver/today
C09 — 部分付款（現收金額 ≠ 應收金額）邊界
填入部分金額，AR 狀態應為 partial，剩餘未收金額正確。
/driver/order/:id → /dayone/ar
C10 — 司機無法看到其他司機的訂單邊界
司機 A 登入後看不到司機 B 的派車任務。
/driver/today

D. 進貨 & 庫存管理
管理員 / 倉管
─
▾
進貨流程
D01 — 建立進貨收貨單
選供應商、填品項數量，送出後出現在進貨列表，狀態為 pending。
/dayone/purchase-receipts
D02 — 供應商簽名 → AP 建立關鍵系統自動
完成供應商簽名後，狀態變 signed，dy_ap_records 新增一筆應付，庫存此時不動。
/dayone/purchase-receipts → /dayone/ar (AP tab)
D03 — 管理員確認入倉 → 庫存增加關鍵系統自動
管理員點確認入倉，狀態變 warehoused，dy_inventory.currentQty 增加，dy_stock_movements 有紀錄。
/dayone/purchase-receipts → /dayone/inventory
庫存 & 回庫流程
D04 — 庫存三段式視圖正確
可賣庫存（currentQty）、待入倉進貨、回庫待驗三欄數字獨立且正確。
/dayone/inventory
D05 — 管理員確認回庫 → 可賣庫存回補關鍵
在回庫待驗列表點確認，dy_pending_returns 狀態更新，dy_inventory.currentQty 增加。
/dayone/inventory
D06 — 手動庫存調整
手動填入原因與絕對值，調整後 dy_inventory_logs 有紀錄，currentQty 正確。
/dayone/inventory
邊界情境
D07 — 供應商簽名後庫存未動（驗規則）邊界
簽名後立即到 /dayone/inventory 確認庫存數字沒有改變，等入倉確認後才動。
/dayone/purchase-receipts → /dayone/inventory
D08 — 庫存不負數保護邊界
列印派車單時庫存不足，currentQty 最低降至 0（GREATEST(0, currentQty - qty)），不出現負數。
/dayone/inventory

一日完整作業流
I01 — 建訂單 → 生成派車單 → 列印有內容關鍵
管理員建訂單、生成派車、列印，確認列印內容完整。
I02 — 司機送達 → AR 應收出現關鍵
司機標記送達，/dayone/ar 的應收立即出現，金額正確。
I03 — 司機收現 → 付款狀態同步關鍵

司機回報剩貨、管理員確認後，dy_inventory.currentQty 增加正確數量。
I05 — 進貨簽收 → AP 出現 → 確認入倉 → 庫存增加關鍵
建進貨單、完成供應商簽名後 AP 出現，管理員確認入倉後庫存增加，AP 金額正確。
LIFF 完整鏈路
I06 — 客戶 LINE 下單 → 派車 → 送達 → AR → 收款關鍵
從 LINE LIFF 下單起，跑完整個流程到帳務收款，每個環節均正確。
