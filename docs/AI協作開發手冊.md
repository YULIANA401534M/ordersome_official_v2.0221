# AI 協作開發手冊

本文件用來規範 `ordersome_official_v2.0221` 專案在 `Claude Code` 與 `Codex` 之間切換時的協作方式，目標是避免上下文斷裂、避免重工、避免資料庫風險。

## 第一原則

每一個新的 AI 開發對話，在開始動手之前，都必須先依序閱讀以下檔案：

1. `CLAUDE.md`
2. `DEVELOPMENT_LOG.md`
3. `todo.md`
4. `CLAUDE.md` 內提到的任務相關文件

不管是 Claude 還是 Codex，都一樣。

如果只是很短的對話，也至少要先讀 `CLAUDE.md`。

## 專案共同記憶來源

不同 AI 之間不能只依賴聊天記憶，真正的共同記憶要落在檔案裡。

以下文件是本專案的共享真相來源：

- `CLAUDE.md`：專案規則、版本、風險、主要優先順序
- `DEVELOPMENT_LOG.md`：已完成事項、日期、版本紀錄
- `todo.md`：接下來要做的事、尚未完成的工作
- `docs/ordersome_module_map_v1.html`：模組完成度地圖
- `docs/ordersome_data_flow_v1.html`：資料流程地圖

只要是重要決策、風險、交接內容，都不能只留在聊天視窗裡。

## 每次開發前的標準流程

每次開始一段新的開發工作時，請先做以下事情：

1. 閱讀 `CLAUDE.md`
2. 查看 `git status`
3. 閱讀 `DEVELOPMENT_LOG.md` 最新區段
4. 確認 `todo.md` 目前主線任務
5. 再開始修改程式

## Claude 與 Codex 並行時的分工原則

當 Claude 與 Codex 同時工作時，必須遵守以下規則：

- 同一時間只能有一個 AI 修改同一個檔案
- 任務切分要以模組為單位，不要切得太零碎
- 不可讓兩個 AI 同時執行 migration
- 不可讓兩個 AI 同時修改 `.env`、部署腳本、資料庫 schema，除非有明確決策

建議分工如下：

- `Claude`：大範圍探索、發想、長篇分析、草擬方案
- `Codex`：結構化實作、除錯、驗證、交接整理

這不是絕對限制，而是預設分工。實際上兩邊都能做前後端，但每一段工作一定要先界定「這次誰負責哪一塊」。

## 資料庫安全規則

本專案涉及 TiDB、商城、ERP、加盟與營運資料，所有資料庫操作都要特別小心。

- 不可在未確認環境的情況下執行破壞性 SQL
- 能先用開發環境或測試資料，就不要直接動正式資料
- 只要有 schema 變更，一定要記錄到 `DEVELOPMENT_LOG.md`
- 新增 migration 時，要順便寫清楚回滾或補救思路
- 不可把 `DATABASE_URL`、API Key、密碼硬寫進程式碼

## AI 交接規則

每次從一個 AI 切換到另一個 AI 之前，至少要更新一份共享文件。

交接最少要包含以下內容：

1. 這次改了什麼
2. 哪些還沒做完
3. 動了哪些檔案
4. 有沒有跑測試
5. 有沒有資料庫或部署風險
6. 下一步最建議做什麼

建議記錄位置：

- 小幅進度更新：`DEVELOPMENT_LOG.md`
- 下一步安排：`todo.md`
- 規則、架構、流程調整：`CLAUDE.md`

## 建議交接格式

可使用以下格式，貼在聊天中或整理進文件：

```md
目前任務：
本次修改檔案：
已完成：
未完成：
風險：
測試結果：
下一步：
```

## 什麼時候應該開新對話

以下情況建議直接開新對話：

- 任務主題已切換到別的模組
- 目前對話太長，資訊已經很亂
- 從規劃階段要切到正式實作
- 前面試了很多做法，想重新整理思路

但在開新對話前，請先把交接內容寫進共享文件。

## 截圖與圖片的使用方式

截圖非常適合拿來討論以下問題：

- UI 畫面錯誤
- RWD 跑版
- 按鈕、彈窗、流程異常
- 商業流程是否符合預期

建議做法：

- 一張截圖只問一個明確問題
- 截圖時要附上頁面名稱與預期結果
- 如果聊天視窗開始不能傳圖，就把重點轉成文字摘要，必要時改開新對話

重要的是：截圖討論完的結論，不要只留在聊天裡，要寫進文件。

## 老闆使用建議

因為專案負責人會在不同 AI 間切換，且不以閱讀程式碼為主，所以每次工作完成後，建議固定向 AI 確認以下內容：

- 這次用白話文做了什麼
- 改了哪些檔案
- 有沒有資料庫風險
- 有沒有部署風險
- 下一步建議做什麼

這樣即使你不看 code，也能掌握節奏。

## 模型使用建議

以本專案目前情況，建議這樣分配：

- `GPT-5.4`：處理高風險問題、架構問題、跨模組複雜除錯
- `GPT-5.3-Codex`：日常主要開發模型
- `GPT-5.4-Mini`：摘要、文件、小修改、低成本討論
- `GPT-5.2`：僅建議用於簡單草稿或一般問答

建議預設順序：

1. 一般開發預設使用 `GPT-5.3-Codex`
2. 碰到資料庫、權限、付款、部署、多模組 bug 時切到 `GPT-5.4`
3. 要省額度、做摘要、整理文件時用 `GPT-5.4-Mini`

## 不可違反規則

只要是本專案新的 AI session，第一個必讀檔案永遠都是 `CLAUDE.md`。
